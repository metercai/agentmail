## Overview

Amail gives your Hermes agent its own email address. Inbound mail becomes a
conversation message, attachments are pre-processed, and multi-party threads
become group chats. Your agent uses **six tools** to send, reply, manage
contacts, and track thread state.

**One command integration.** `bash integrate.sh` auto-detects your gateway,
validates credentials, saves config, and runs diagnostics — all in one pass.

---

## Architecture

```
┌─────────────┐     SMTP      ┌──────────────┐    POST /webhooks/amail-inbound    ┌────────────────┐
│  External    │ ────────────→ │  amail-gateway │ ─────────────────────────────────→ │ Hermes Gateway │
│  (email)     │               │  (Rust)      │   X-Webhook-Signature (HMAC)       │ (webhook platform)│
└─────────────┘               └──────────────┘                                    └──────┬─────────┘
                                                                                         │
                                                                    ┌────────────────────┤
                                                                    │ preprocessor       │
                                                                    │ → sender display   │
                                                                    │ → persona parse    │
                                                                    │ → direct_message   │
                                                                    │ → attachment fetch │
                                                                    ├────────────────────┤
                                                                    │ store_inbound_msg  │
                                                                    │ → agent_state(msg) │
                                                                    │ → raw_email snap   │
                                                                    ├────────────────────┤
                                                                    │ /amail skill load  │
                                                                    │ → agent processes  │
                                                                    └────────────────────┘
```

**Key design decisions:**
- Webhook secret is **gateway-managed** (Hermes creates it, gateway receives it)
- All agent addresses share one route: `amail-inbound`
- Persona is encoded in the email address: `persona.profile@domain`

## Quick Start (Script)

```bash
# From amail integrations/hermes directory
bash integrate.sh

# 10-step wizard:
#   [1/10] gateway connect     — auto-detected from localhost:38080
#   [2/10] auth method       — enter admin_key or product code
#   [3/10] domain            — select or enter agent email domain (admin_key only)
#   [4/10] basic config      — save snapshots? + default manager email address
#   [5/10] save config       — writes ~/.hermes/amail_gateway.json (or activates system)
#   [6/10] install tools     — copies amail_tools.py + registers in toolsets.py
#   [7/10] patch webhook     — injects preprocessor support
#   [8/10] patch profiles    — injects profile lifecycle hooks
#   [9/10] diagnostics       — verify_integration() checks
#   [10/10] send/receive test — creates test key, sends email, cleans up

# Select language: English (default) or Chinese
# Non-interactive (CI/scripts):
AMAIL_GATEWAY_URL=http://gateway:38080 \
AMAIL_ADMIN_KEY=sk-... \
AMAIL_DOMAIN=admin.local \
AMAIL_SAVE_SNAPSHOTS=false \
AMAIL_MANAGER_ADDRESS=admin@admin.local \
  bash integrate.sh --auto

# Product code path (new system):
AMAIL_GATEWAY_URL=http://gateway:38080 \
AMAIL_PRODUCT_CODE=prod-xxxx-xxxx-xxxx \
AMAIL_MANAGER_ADDRESS=admin@example.com \
  bash integrate.sh --auto

# Docker / custom network: override webhook callback host
AMAIL_WEBHOOK_HOST=192.168.1.100 bash integrate.sh --auto
```

**Two authentication paths:**
| Path | What you need | What happens |
|---|---|---|
| admin_key | Existing admin API key | Config saved directly |
| product_code | Pre-generated activation code | System activated, admin_key returned |

The product_code path auto-skips Step 3 (domain is generated by the server).

**Environment variables:**

| Variable | Required | Default | Purpose |
|---|---|---|---|
| `AMAIL_GATEWAY_URL` | yes (auto) | — | Gateway server URL |
| `AMAIL_ADMIN_KEY` | admin_key path | — | Existing system admin API key |
| `AMAIL_PRODUCT_CODE` | product_code path | — | One-time product activation code |
| `AMAIL_DOMAIN` | no | empty | Agent email domain |
| `AMAIL_SAVE_SNAPSHOTS` | no | false | Save raw email snapshots locally |
| `AMAIL_MANAGER_ADDRESS` | no | empty | Default manager email for agents |
| `AMAIL_LANG` | no | en | Language selection (`en` or `zh`) |
| `AMAIL_WEBHOOK_HOST` | no | auto-detected | Override webhook callback host (Docker, custom networks) |
| `AMAIL_BRIDGE_URL` | no | empty | Full bridge URL for push mode (e.g. `https://bridge.example.com/webhooks/amail-inbound`) |

The script handles config, tool installation, source patches, diagnostics, and a send/receive test — everything needed to get amail working with Hermes.

## Agent Profile Lifecycle

### 1. Profile Creation

When a Hermes profile is created, the **profile hook** fires automatically:

1. Registers `{profile_name}@{domain}` as a webhook route on the gateway
2. Sets the **manager address** in domain metadata
3. Auto-whitelists the manager so the agent can send them contact requests
4. Creates the `amail-inbound` webhook route on the gateway (idempotent)
5. Generates an **address activation code** and saves it to the profile

```json
// {profile_dir}/amail.json — written automatically
{
  "email": "alice@admin.local",
  "activation_code": "addr-xxxx-xxxx-xxxx-xxxx-xxxx-xxxx",
  "gateway_url": "http://localhost:38080",
  "domain": "admin.local",
  "system_id": "admin",
  "manager_address": "admin@admin.local",
  "save_raw_snapshots": false,
  "webhook_host": "127.0.0.1",
  "bridge_url": "https://bridge.example.com/webhooks/amail-inbound"
}
```

> **Security**: The profile config contains an `activation_code`, NOT the raw
> `api_key`. The admin process never sees the agent's API key.

### 2. Agent Startup

At agent startup, `agent_startup_activate()` detects the `activation_code` and
exchanges it for a real `api_key` via `POST /api/v1/activate-address`.
The `api_key` is written back to `amail.json` and the `activation_code` is removed.

### 3. Profile Deletion

When a profile is deleted, the agent's API key is removed from the gateway and
`amail.json` is deleted. Domain registrations and webhook routes are preserved
for future re-activation.

## Persona Switching

Amail supports dynamic persona switching via the email address format:
`persona.profile@domain`. Each profile can define multiple personas in its
own `config.yaml`:

```yaml
# {profile_dir}/config.yaml
agent:
  personalities:
    support: "You are a helpful support agent. Be patient and empathetic."
    sales:   "You are a persuasive sales agent. Focus on value propositions."
```

**How it works:**

1. Someone sends email to `support.alice@admin.local`
2. The preprocessor strips the persona prefix and validates it against the profile's configured `agent.personalities`
3. If valid, the agent sees `my_amail_addr = "support.alice@admin.local"` and adopts the support persona
4. If the persona is not configured, the address falls back to `alice@admin.local` (warning logged)

Personas are **per-profile** — each profile validates against its own config.
No cross-profile persona leakage. Persona configuration is managed by the
Hermes user, not by the integration script.

## Inbound Message Model

Each inbound email arrives as a JSON message. The preprocessor produces these
fields; additional fields are injected by the gateway pipeline.

| Field | Source | Meaning |
|-------|--------|---------|
| `message_id` | gateway SMTP | Unique identifier for threading |
| `subject` | gateway SMTP | Email subject |
| `body` | gateway SMTP | Plain text body |
| `sender` | preprocessor | `Name <email>` format — the person talking to you |
| `sender_profile` | gateway | Contact profile of sender (auto-populated) |
| `recipients` | preprocessor | `{to: [...], cc: [...]}` — everyone on the thread |
| `recipients_profile` | gateway | Contact profiles for all recipients except you |
| `my_amail_addr` | preprocessor | Your persona-aware address (e.g. `support.alice@domain`) |
| `my_profile` | gateway | Your own contact profile |
| `direct_message` | preprocessor | `true` = you're the only `to` recipient (no CC) |
| `mentioned` | preprocessor | Someone wrote `@your-name` in the body |
| `thread_summary` | gateway | Previous thread state from `set_email_summary` |
| `attachments` | preprocessor | Local file paths (DOCX/XLSX/PDF get `.md` extracts) |

## Inbound Processing Pipeline

```
SMTP → gateway
  → resolve system_domains (exact match → bare domain fallback)
  → HMAC-SHA256 sign → X-Webhook-Signature
  → POST /webhooks/amail-inbound
    ├─ HMAC validation (gateway)
    ├─ preprocess_mail_payload(payload, headers)
    │   ├─ Extract display names from MIME headers → sender field
    │   ├─ Parse persona from 'to' address (persona.profile@domain)
    │   ├─ Set recipients: {to: [...], cc: [...]}
    │   ├─ Set my_amail_addr: persona-aware agent address
    │   ├─ Compute direct_message: single to-recipient + no CC
    │   ├─ Compute mentioned: @name in body
    │   └─ Download attachments → local cache (.md extracts for docs)
    ├─ store_inbound_message()
    │   ├─ msg:{mid} metadata → gateway agent_state
    │   └─ save_raw_snapshots=true → raw_email/{agent}/{yyyymm}/in-{mid}.json
    ├─ Inject sender_profile / recipients_profile / my_profile / thread_summary
    ├─ /personality {persona} (if route has persona field)
    ├─ /amail skill invocation
    └─ Agent processes email → uses send_mail tool to reply
```

## Agent Tools (6 tools, toolset: `amail`)

| Tool | Purpose |
|------|---------|
| `send_mail(to, subject, body, message_id?, cc?, attachments?)` | Send or reply to emails. Pass `message_id` for threading. |
| `contact_profile(address?, name?)` | Look up a contact by address or name |
| `set_contact_profile(address, description)` | Store/update contact profile (JSON merge, auto name-index) |
| `manage_contacts(action, address, direction?)` | Check, add, or remove contacts from whitelist |
| `email_summary(message_id)` | Retrieve stored thread summary |
| `set_email_summary(message_id, summary)` | Save updated thread state after replying |

## Integration Functions

| Function | Purpose |
|----------|---------|
| `setup(gateway_url, system_id, admin_key?, product_code?, domain?, save_raw_snapshots?, manager_address?)` | Save gateway config to `~/.hermes/config.yaml` |

| `parse_amail_persona(email)` | Extract (persona, profile) from `persona.profile@domain` |

## Security Model

| Credential | Who Has It | Purpose |
|------------|-----------|---------|
| Admin key (`system` scope) | Gateway operator, `integrate.sh` | Manage domains, generate codes, configure whitelists |
| Address activation code | Profile config (on disk) | One-time: agent self-activates → returns raw api_key |
| Agent API key (`agent` scope) | Profile config (on disk) | Send/receive mail, manage contacts, write thread summaries |
| Webhook HMAC secret | Gateway | System-level trust between gateway and Hermes gateway |

**Key principles:**
- Activation code creator NEVER sees the resulting raw API key
- Each agent has its own API key with minimal scopes
- Activation codes are one-time use
- Manager ↔ Agent whitelist is created automatically during profile setup

## Configuration Reference

### Gateway Config (`~/.hermes/amail_gateway.json`)

Standalone file — does NOT pollute `config.yaml`:

```json
{
  "gateway_url": "http://localhost:38080",
  "admin_key": "sk-...",
  "system_id": "admin",
  "domain": "admin.local",
  "save_raw_snapshots": false,
  "manager_address": "admin@admin.local",
  "webhook_host": "127.0.0.1"
}
```

The `webhook_host` field is **auto-detected** during integration by comparing
gateway_url against local network interfaces. It determines the address the
gateway uses to POST webhook callbacks to the Hermes gateway:

| Gateway location | Detected `webhook_host` |
|---|---|
| Same machine | `127.0.0.1` |
| Same LAN | Our LAN IP |
| Public internet | External IP (or LAN fallback) |
| Docker (manual) | Set `AMAIL_WEBHOOK_HOST` env var |

Override with `AMAIL_WEBHOOK_HOST` env var or edit `amail_gateway.json` directly.

**Optional `bridge_url` — amail-bridge push mode:**

When using the [amail-bridge](../docs/amail-bridge-design.md) push mode, add
`bridge_url` to route all webhook callbacks through the bridge:

```json
{
  "bridge_url": "https://bridge.example.com/webhooks/amail-inbound"
}
```

If `bridge_url` is present, `webhook_host` is ignored. The bridge handles
TLS termination and internal routing to per-profile gateway ports.

**Optional `delivery_mode` — amail-bridge pull mode:**

When using the bridge pull mode (no public IP), set per-domain on the gateway
via `POST /api/v1/admin/systems/{sid}/domains` with `delivery_mode: "pull"`.
SMTP mail will be stored in gateway pending queue instead of POSTed to webhook.

### Profile Config (`{profile_dir}/amail.json`)

```json
{
  "gateway_url": "http://localhost:38080",
  "api_key": "sk-...",
  "email": "alice@admin.local",
```

## Storage

| Location | Key | Content |
|----------|-----|---------|
| gateway `agent_state` | `profile:{address}` | Contact profile JSON |
| gateway `agent_state` | `name:{name}` | `{"addresses": [...]}` index |
| gateway `agent_state` | `thread:{thread_id}` | Thread summary text |
| gateway `agent_state` | `msg:{message_id}` | `{"references": [...], "thread_id": "..."}` |
| Local (optional) | `raw_email/{agent}/{yyyymm}/in-{mid}.json` | Agent-visible inbound snapshot |
| Local (optional) | `raw_email/{agent}/{yyyymm}/out-{mid}.json` | Agent-composed outbound snapshot |

> **Snapshots**: When `save_raw_snapshots: true` in profile config, the
> agent-visible JSON (post-preprocessing) is saved for both inbound and outbound
> messages. These are the exact payloads the agent works with — useful for
> debugging and replay.

## HMAC Compatibility

Gateway signs webhook payloads with `X-Webhook-Signature: {HMAC-SHA256(body, secret)}`.
This is the generic format recognized by Hermes' webhook adapter.

The secret is shared between the Hermes gateway (which generates it) and the
gateway (which receives it per-agent during domain registration).
